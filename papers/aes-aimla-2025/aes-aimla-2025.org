* Introduction

Differentiable programming via automatic differentiation (AD) is the
bedrock upon which gradient-based mathematical optimisation techniques
lie, and forms the basis for current approaches to machine
learning. Differentiability of the primitive operators of a
programming language permits the creation of programs that are
end-to-end differentiable with respect to their input variables;
coupled with a loss function --- also composed of differentiable
primitives --- such a program can produce gradients, which, when
backpropagated as update-values to the input variables, cause those
inputs to approach target values, reducing the value returned by the
loss function, etc., etc. Differentiable digital signal processing
(DDSP) is concerned with the creation and application of
differentiable signal processors 

Having previously demontstrated that DDSP is possible in the Faust
language, here we extend to a more general AD syntax and an
accompanying set of programmatic conventions. We show that IIR filters
can be made differentiable in Faust and their coefficients optimised
via gradient descent. We also demonstrate that neural network
structures can be created in a composable, extensible fashion in
Faust, pointing the way toward an audio-focused ecosystem for the
development of solutions to problems based on machine learning
approaches. Though imposing a particular syntactic approach,
dispensing with support for certain features of the Faust language, we
contend that such restrictions are comparable with those posed by
established automatic differentiation and machine learning
frameworks.

Faust is a functional paradigm, domain specific language (DSL) for
audio digital signal processing (DSP) [cite:@orlareyFAUST2009]. It is
built atop a domain-agnostic /block diagram algebra/ (BDA)
[cite:@orlareyAlgebra2002] that describes primitive operations (though
the nature of these operations is not important at the level of the
BDA), connected into graphs via a selection of /composition
operators/. A textual language, Faust can, for example, define a
diagram consisting of two blocks --- \(A\), with \(i_{A}\) inputs and
\(o_{A}\) outputs, \(B\), with \(i_{B}\) inputs and \(o_{B}\) outputs, \(i_{A} =
o_{A} = i_{B} = o_{B} = 1\) --- composed in sequence, with code of the form
#+name: lst:seq

#+begin_src faust :noexec

#+end_src

Conceptually, a given block diagram, \(D\) (see cref:fig:seq for its
graphical depiction) may undergo semantic transformation, notated
\(\llbracket \cdot \rrbracket\), into a signal processor
[cite:@orlareySyntactical2004]:
#+NAME: eq:semanticTransformation
\begin{equation}
  \begin{split}
    \llbracket D \rrbracket &\;:\; \mathbb{S}^{n} \to \mathbb{S}^{m} \\
    \llbracket D \rrbracket(s_{1},\dots,s_{n}) &= (y_{1},\dots,y_{m}) \\
    \mathbf{y}(t) &= h(\mathbf{s}(t))\;,
  \end{split}
\end{equation}
where \(\mathbb{S}\) is the set of all signals, \(h\) is a
mathematical function analogous to the operations and compositions of
\(D\), and \(t\) is time. The signal processor receives a vector of
input signals, \(\mathbf{s}\), and produces a vector of output signals
\(\mathbf{y}\).

** The Faust Ecosystem

Faust is a highly optimised, compiled language [cite:@orlareyNew2020],
with C++ as its default compilation target. One of the great strengths
of the Faust ecosystem is the provision of support for a variety of
additional backends, including LLVM [cite:@letzOverview2018],
WebAssembly [cite:@letzCompiling2017], and Rust
[cite:@gasterAudio2020], plus numerous software and hardware
platforms. Faust code can be compiled for use in VST
plugins[cite:@albouyFaust2017], for embedded systems
[cite:@michonEmbedded2020] and FPGAs [cite:@cochardProgramming2024];
it is used by artists and digital luthiers
[cite:@michonFaust2020a;@rushtonSpringamajig2022;@hansenWeather2018],
and features sophisticated libraries for wave digital filter modelling
[cite:@roosenburgWave2021] and physical modelling synthesis
[cite:@russoIntroducing2021;@michonFaust2018], amongst others.

Just as Python-based machine learning frameworks such as PyTorch and
JAX permit users to perform automatic differentiation on arbitrary
code, the same should ideally be true of an equivalent Faust-based
framework. [So, actually this isn't entirely the case; one has to
write code in a particular style for JAX/PyTorch to handle it.]

Automatic differentiation at the level of the Faust language is
predicated on the use of Faust's /pattern-matching/ abstraction;
pattern-matching can be thought of as something akin to the operator
overloading approach that is possible in languages such as Python and
C++. A pattern-matching definition is indistinguishable from that of a
function, but exlpoits the fact that a function may receive, as a
parameter, a programmatic expression. For example a forward-mode
differentiable addition primitive, acting on dual signals \(U = \langle{}u,
\nabla{}u\rangle, V = \langle{}v, \nabla{}v\rangle\), may be created by defining a function that accepts
the addition operator as a parameter, and produces, in parallel, the
addition primitive's primal output, \(u+v\), and its gradients,
\(\nabla{}u+\nabla{}v\):

#+name: lst:diffAdd
#+caption: Definition of a differentiable addition primitive in the
#+caption: Faust language.
#+begin_src faust :noeval
diff(+) = interleave(vars.N + 1, 2)
    : +,par(i, vars.N, +);
#+end_src

Here, the function ~interleave~ ensures that the primal signals and
their respective partial derivatives are delivered to the addition
operations in the appropriate order: \(u, v, \frac{\partial{}u}{\partial{}x_{1}},
\frac{\partial{}v}{\partial{}x_{1}},\dots,\frac{\partial{}u}{\partial{}x_{N}}, \frac{\partial{}v}{\partial{}x_{N}}\) (see
cref:fig:diffadd).

Faust is based on a /block-diagram algebra/
[cite:@orlareySyntactical2004]. In this algebra, a given block
diagram, \(D\), is comprised of a primitive block, \(P\), or the
composition of two block diagrams via a composition operator according
to the syntactic rule[fn:1]
#+NAME: eq:syntax
\begin{equation}
D = P \;|\; (D_{1} \circ D_{2})\;,
\end{equation}
where \(\circ\) is one of the sequential (:), parallel (,), split (<:),
merge (:>), or recursive (\(\sim\)) composition operators. The creation
of differentiable Faust programs has been shown to be possible the
composition of differentiated primitive blocks
[cite:@rushtonDifferentiable2024]; using \(\Delta(\cdot)\) to denote derivative
transformation, the syntactic rule for differentiation in Faust was
#+NAME: eq:badDiffSyntax
\begin{equation}
\Delta(D) = \Delta(P) \circ \Delta(P) \circ \dots \circ \Delta(P)\;.
\end{equation}
While effective in facilitating differentiable programming, this
syntax imposed an undesirable verbosity. Our preferred syntactic rule
should be a direct transformation of cref:eq:syntax:
#+NAME: eq:idealDiffSyntax
\begin{equation}
\Delta(D) = \Delta(P) \;|\; \Delta(D_{1} \circ D_{2}) \;.
\end{equation}
The ideal outcome is to permit the programmer to subject an entire
Faust program to automatic differentiation in a single expression,
minimising the amount of re-writing required to ready their code for
transformation. This would render a great proportion of existing Faust
code differentiable, and, in doing so, prepare that wealth of material
for prospective use in optimisation problems, and machine learning
applications. In order to achieve such an outcome, it is necessary to
provide differentiation rules for the five composition operators,
taking into account their arithmetical significance, if any, in a
block diagram to which they contribute.

** Differentiable Block-Diagram Compositions

To define syntactic rules for the derivatives of Faust's composition
operators entails analysing them as if they were themselves primitive
blocks. To do this, we must first consider three of Faust's
fundamental primitive operators for their differentiation rules; their
derivatives will assist us in achieving differentiable compositions.

*** Identity Primitive

Faust's identity function, notated \(\_\), is itself a primitive block,
and in turn describes a connection between two blocks. The signal
processor represented by the identity function adheres to the
following semantic scheme [cite:@orlareySyntactical2004]:
#+NAME: eq:identity
\begin{equation}
  \begin{split}
    \llbracket \_ \rrbracket &\;:\; \mathbb{S} \to \mathbb{S} \\
    \llbracket \_ \rrbracket(s) &= (s)\;,
  \end{split}
\end{equation}
which is to say that it receives one signal, \(s\), and returns that
same signal unchanged. Application of the identity function is
analogous to multiplying \(s(t)\), a function of time, by a constant
signal of value \(1\); supposing that \(s(t)\) is dependent on some
input variable \(x\), its partial derivative with respect to \(x\) is
#+NAME: eq:diffsig
\begin{equation}
\frac{\partial{}}{\partial{}x}(1 \cdot s(x, t)) = \frac{\partial{}}{\partial{}x}s(x, t)\;,
\end{equation}
and the semantic rule for the differentiable identity function is thus
#+NAME: eq:diffidentity
\begin{equation}
  \begin{split}
    \llbracket \Delta(\_) \rrbracket &\;:\; \mathbb{S}_{d} \to \mathbb{S}_{d} \\
    \llbracket \Delta(\_) \rrbracket(\langle{}s, \nabla{}s\rangle) &= (\langle{}s, \nabla{}s\rangle)\;.
  \end{split}
\end{equation}

*** Cut Primitive

The cut function (!), terminates a signal; intuitively, the
derivative of a termination is itself a termination:
#+NAME: eq:cut
\begin{equation}
  \begin{split}
    \llbracket \Delta(!) \rrbracket &\;:\; \mathbb{S}_{d} \to \mathbb{S}^{0} \\
    \llbracket \Delta(!) \rrbracket(\langle{}s,\nabla{}s\rangle) &= ()\;.
  \end{split}
\end{equation}
The circumstances under which it is useful to compute a signal path
only to discard it may at first appear obscure, until one considers
that combinations of the identity and cut functions can be used to
route signals between blocks [cite:@orlareySyntactical2004]. For
instance, interleave two groups of two signals, such that
\(s_{1,1},s_{1,2},s_{2,1},s_{2,2}\) becomes \(s_{1,1},s_{2,1},s_{1,2},s_{2,2}\), one may write
#+NAME: eq:interleave22
\begin{equation}
\_,\_,\_,\_ \;<:\; \_,!,\_,!,!,\_,!,\_\;.
\end{equation}
The general interleaving function, \(\chi(r,c)\), which interleaves \(c\)
groups of \(r\) signals, described in cref:fig:ad-rules, is used in
cref:lst:diffAdd; here is its definition in code:
#+begin_src faust :results file svg :file images/interleave.svg :exports code :noeval
cutpq(0, q) = _,par(l, q, !);
cutpq(p, 0) = par(l, p, !),_;
cutpq(p, q) =
  par(l, p, !),cutpq(0, q);

interleave(R, C) = par(l, R * C, _)
  <: par(m, R, par(n, C,
       cutpq(m, R - m - 1)));
#+end_src
In principle, being constructed from the differentiable identity and
cut primitives, \(\chi\) should in turn be differentiable. This is
contingent, however, on the derivative behaviour of the split
composition operator; see cref:sec:splitComposition for further
detail.

*** Delay Primitive

Faust in fact provides two primitives for delaying signals; here we
will focus on the ~mem~ primitive, which represents a one-sample
delay, and can be applied repeatedly in sequence to introduce delays
of longer duration. In general, a fixed delay can be represented as
the convolution of a signal, \(s(x,t)\), with a time-shifted impulse,
#+NAME: eq:delayAsConv
\begin{equation}
s(x, t-\tau) = s(x, t) \ast h(t)\;,\quad h(t) = \delta(t - \tau)\;.
\end{equation}
Taking the temporal-parametric Fourier transform of the derivative
with respect to \(x\) of the convolution, using the convolution and
differentiation properties of the Fourier transform
[cite:@mcclellanSignal2003], yields
#+NAME: eq:
\begin{align}
&\quad\;\mathcal{F}\left[\frac{\partial{}}{\partial{}x}\left(s(x,t) \ast h(t)\right)\right] \nonumber\\
  &= j\omega_{x}S(j\omega_{x},j\omega_{t})H(j\omega_{t}) \\
  &= \mathcal{F}\left[\frac{\partial{}}{\partial{}x}s(x,t)\right]\cdot\mathcal{F}\left[h(t)\right]\;,
\end{align}
which demontsrates that differentiation is time-invariant, and means
that we can use the following semantic scheme for the differentiable
~mem~ primitive:
#+NAME: eq:diffmem
\begin{equation}
  \begin{split}
    \llbracket \Delta(\texttt{mem}) \rrbracket &\;:\; \mathbb{S}_{d} \to \mathbb{S}_{d} \\
    \llbracket \Delta(\texttt{mem}) \rrbracket(\langle{}s, \nabla{}s\rangle) &= (\langle{}y, \nabla{}y\rangle) \\
    \langle{}y,\nabla{}y\rangle(t) &= \langle{}s,\nabla{}s\rangle(t-1)\;.
  \end{split}
\end{equation}

*** Sequential Composition

A mathematical analogy to the composition of blocks in sequence is
function composition. Assuming an input variable, \(x\), a suitable
mathematical expression for the program in cref:lst:seq is
#+NAME: eq:composition
\begin{equation}
y = (B \circ A)(x)\;.
\end{equation}
In Faust, for \(A\) and \(B\) to be composable in this manner, \(o_{A}\)
must be equal to \(i_{B}\).[fn:2] Consequently, the following block
diagram expressions are arithmetically equivalent
\begin{gather}
D_{1} = A : B \label{eq:seq} \\
D_{2} = A : \Big\|_{l=1}^{o_{A}}\_ : B \label{eq:seqViaWire};\,
\end{gather}
and, letting \(o_{A} = i_{B} = 1\), it is clear that the following program
is equivalent to the one depicted in cref:fig:seq:
# #+name: lst:seqWithWire
# #+caption: Sequential composition of two Faust blocks interposed with
# #+caption: an identity block.
#+begin_src faust :noeval
process = A : _ : B;
#+end_src
The function composition \((B \circ I \circ A)(x)\), where \(I\) is the
identity function, simplifies to the expression in
cref:eq:composition.  Using this fact, cref:eq:seq,eq:seqViaWire, and
the semantic scheme described in cref:eq:diffidentity,
#+NAME: eq:diffSeq
\begin{align}
\Delta(A : B) &= \Delta(A) : \big\|_{l=1}^{o_{A}}\Delta(\_) : \Delta(B) \nonumber \\
  &= \Delta(A) : \Delta(B)\;.
\end{align}

*** Parallel Composition

Blocks composed in parallel share no common operations, and do not
influence one-another, arithemtically-speaking:

#+begin_src faust :noexec
process = A,B;

*** Recursive Composition

It is a long-established fact that recursive filters are
differentiable [cite:@shynkAdaptive1989]. The derivative of the
\(N\)-pole filter
#+NAME: eq:all-pole
\begin{equation}
y(t) = s(t) + \sum_{l=1}^{N}x_{l}(t)y(t-l)
\end{equation}
with respect to coefficient \(x_{l}\) is itself a recursion:
#+NAME: eq:all-pole-diff
\begin{equation}
\frac{\partial{}y(t)}{\partial{}x_{l}(t)} = y(t - l) + \sum_{m=1}^{N}x_{m}(t)\frac{\partial{}y(t-m)}{\partial{}x_{l}(t)}\;.
\end{equation}
Recursive composition in Faust, \(A \sim B\), introduces an implicit
one-sample delay, indicated by the small square following block \(A\)
in cref:fig:rec. Substituting an addition for \(A\) (\(i_{A} = 2\)) and
multiplication by a coefficient, \(x\), for \(B\) (\(i_{B} = 1\))
produces a one-pole filter whose derivative, after
cref:eq:all-pole-diff, is
#+NAME: eq:one-pole-diff
\begin{equation}
\frac{dy(t)}{dx(t)} = y(t - 1) + x(t)\frac{dy(t-1)}{dx(t)}\;.
\end{equation}
