* Introduction

Differentiable programming via automatic differentiation (AD) is the
bedrock upon which gradient-based mathematical optimisation techniques
lie, and forms the basis for current approaches to machine
learning. Differentiability of the primitive operators of a
programming language permits the creation of programs that are
end-to-end differentiable with respect to their input variables;
coupled with a loss function --- also composed of differentiable
primitives --- such a program can produce gradients, which, when
backpropagated as update-values to the input variables, cause those
inputs to approach target values, reducing the value returned by the
loss function, etc., etc. Differentiable digital signal processing
(DDSP) is concerned with the creation and application of
differentiable signal processors 

Having previously demontstrated that DDSP is possible in the Faust
language, here we extend to a more general AD syntax and an
accompanying set of programmatic conventions. We show that IIR filters
can be made differentiable in Faust and their coefficients optimised
via gradient descent. We also demonstrate that neural network
structures can be created in a composable, extensible fashion in
Faust, pointing the way toward an audio-focused ecosystem for the
development of solutions to problems based on machine learning
approaches. Though imposing a particular syntactic approach,
dispensing with support for certain features of the Faust language, we
contend that such restrictions are comparable with those posed by
established automatic differentiation and machine learning
frameworks.

Faust is a functional paradigm, domain specific language (DSL) for
audio digital signal processing (DSP) [cite:@orlareyFAUST2009]. It is
built atop a domain-agnostic /block diagram algebra/ (BDA)
[cite:@orlareyAlgebra2002] that describes primitive operations (though
the nature of these operations is not important at the level of the
BDA), connected into graphs via a selection of /composition operators/
(see cref:fig:compositions). A textual language, Faust can, for
example, define a diagram consisting of two blocks, \(A\) and \(B\),
composed in sequence, with code of the form
#+name: lst:seq
#+caption: Sequential compostion of two Faust blocks.
#+begin_src faust :noexec
process = A : B;
#+end_src
Conceptually, a given block diagram, \(D\), may undergo semantic
transformation, notated \(\llbracket \cdot \rrbracket\), into a signal
processor [cite:@orlareySyntactical2004]:
#+NAME: eq:semanticTransformation
\begin{equation}
  \begin{split}
    \llbracket D \rrbracket &\;:\; \mathbb{S}^{n} \to \mathbb{S}^{m} \\
    \llbracket D \rrbracket(s_{1},\dots,s_{n}) &= (y_{1},\dots,y_{m}) \\
    \mathbf{y}(t) &= h(\mathbf{s}(t))\;,
  \end{split}
\end{equation}
where \(\mathbb{S}\) is the set of all signals, \(h\) is a
mathematical function analogous to the primitive operations and
compositions of \(D\), and \(t\) is time. The signal processor
receives a vector of input signals, \(\mathbf{s}\), and produces a
vector of output signals \(\mathbf{y}\).

** The Faust Ecosystem

Faust is a highly optimised, compiled language [cite:@orlareyNew2020],
with C++ as its default compilation target. One of the great strengths
of the Faust ecosystem is its /write once, run anywhere/ model; a
variety of additional backends are supported, including LLVM
[cite:@letzOverview2018], WebAssembly [cite:@letzCompiling2017], and
Rust [cite:@gasterAudio2020], plus numerous software and hardware
platforms. Faust code can be compiled for use in VST
plugins[cite:@albouyFaust2017], for embedded systems
[cite:@michonEmbedded2020] and FPGAs [cite:@cochardProgramming2024];
it is used by artists and digital luthiers
[cite:@michonFaust2020a;@rushtonSpringamajig2022;@hansenWeather2018],
and features sophisticated libraries for wave digital filter modelling
[cite:@roosenburgWave2021] and physical modelling synthesis
[cite:@russoIntroducing2021;@michonFaust2018], amongst others.

** Differentiable Programming in Faust

In light of Faust's versatility, efficiency, and strengths as a DSL in
tackling audio problems it is little suprise that it has been turned
to machine learning problems in the audio domain, via /DawDreamer/
[cite:@braunDawDreamer2021], which uses a Python backend to transpile
Faust code for gradient-based optimisation with the JAX Python
library. JAX, and other well-known machine learning frameworks, such
as TensorFlow and PyTorch, enable gradient-based optimisation
approaches by way of /automatic differentiation/ (AD), a technique for
computing the numerical outputs of both a mathematical expression (or,
more commonly, a computer program) and its derivatives in parallel
[cite:@rallArithmetic1986;@baydinAutomatic2018;@harrisonBrief2021].

/Differentiable programming/ [cite:@wangDemystifying2019] is, broadly
speaking, the creation of computer programs using primitive operations
which are themselves differentiable; a computer program consisting of
differentiable primitives may be differentiated automatically,
producing the desired /primal/ and /gradient/ numerical outputs for
use in a gradient-based optimisation algorithm. The feasibility of
differentiable programming, and thus differentiable digital signal
processing (DDSP), in the Faust language has been demonstrated via the
implementation of forward-mode AD primitives
[cite:@rushtonDifferentiable2024].

One way of framing forward mode AD is as an arithemitic of dual
numbers [cite:@rallArithmetic1986; @yuDNAD2013] of the form
\(\langle{}u,\nabla{}u\rangle\), where \(u\) is the aforementioned primal, and \(\nabla{}u
= \begin{bmatrix}\frac{\partial{}u}{\partial{}x_{1}} & \cdots & \frac{\partial{}u}{\partial{}x_{N}}\end{bmatrix}^{T}\)
the gradient of \(u\), its partial derivatives with to a vector of
\(N\) input variables, \(\mathbf{x}\). A dual number can be passed
into a forward-mode differentiable primtive, \(f\), yielding an output
dual number, \(\langle{}f(u),\nabla{}uf'(u)\rangle\). Faust is a language of /signals/ and
signal processors; consequently we will refer to the /dual signal/
\(\langle{}s,\nabla{}s\rangle \in \mathbb{S}_{d}\), where \(\mathbb{S}_{d} = \mathbb{S}^{N+1}\). The
signal processor based on the differentiated block diagram \(D\) has
the following semantic scheme:
#+NAME: eq:diffSemantic
\begin{equation}
\begin{split}
\Delta_{N}\llbracket D \rrbracket &\;:\; \mathbb{S}_{d}^{M} \to \mathbb{S}_{d}^{P} \\
\Delta_{N}\llbracket D \rrbracket(\langle{}s_{1},\nabla{}s_{1}\rangle,\dots&,\langle{}s_{M},\nabla{}s_{M}\rangle) \\
&= (\langle{}y_{1},\nabla{}y_{1}\rangle,\dots,\langle{}y_{P},\nabla{}y_{P}\rangle) \\
\langle\mathbf{y}(t),\mathbf{J}_{\mathbf{y}}(t)\rangle &= \langle{}h(\mathbf{s}(t)), \mathbf{J}_{{\mathbf{s}}}(t)\nabla{}h(\mathbf{s}(t))\rangle\;,
\end{split}
\end{equation}
where \(\Delta_{N}\llbracket \cdot \rrbracket\) indicates derivative
transformation with respect to \(N\) input variables, and
\(\mathbf{J}_{\mathbf{s}}\) and \(\mathbf{J}_{\mathbf{y}}\) are,
respectively, \(M\times{}N\) and \(P\times{}N\) Jacobian matrices.

AD primitives have been demonstrated in Faust using the language's
pattern-matching abstraction
[cite:@rushtonDifferentiable2024]. Pattern-matching can be thought of
as something akin to the operator overloading approach that is
possible in languages such as Python and C++, and exploits the fact
that a Faust function definition may receive, as a parameter, a
programmatic expression. For example, a differentiable addition
primitive, acting on dual signals \(\langle{}s_{1}, \nabla{}s_{1}\rangle,\langle{}s_{2}, \nabla{}s_{2}\rangle\), may be
created by defining a function that accepts the addition operator as a
parameter, and produces, in parallel, the addition primitive's primal
output, \(s_{1}+s_{2}\), and its gradients, \(\nabla{}s_{1}+\nabla{}s_{2}\) (see cref:lst:diffAdd).
#+name: lst:diffAdd
#+caption: Definition of a differentiable addition primitive in the
#+caption: Faust language.
#+attr_latex: :float t 
#+begin_src faust :noeval
diff(+) = interleave(N + 1, 2)
    : +,par(i, N, +);
#+end_src
The function ~interleave~ ensures that the primal signals and their
respective partial derivatives are delivered to the addition
operations in the appropriate order: \(s_{1}, s_{2}, \frac{\partial{}s_{1}}{\partial{}x_{1}},
\frac{\partial{}s_{2}}{\partial{}x_{1}},\dots,\frac{\partial{}s_{1}}{\partial{}x_{N}}, \frac{\partial{}s_{2}}{\partial{}x_{N}}\), as illustrated
in cref:fig:diffadd. One addition primitive is created for the primal
operation, and a further \(N\) additions in parallel for computing
partial derivatives, which we can notate \(\big\|_{l=1}^{N} +\).

** Motivation

Considering the plethora of existing approaches to DDSP
[cite:@hayesReview2023], why pursue an implementation in Faust? The
existence of DawDreamer would appear to obviate such an endeavour,
until one considers that the AD transformations facilitated by JAX lie
the wrong side, as it were, of the Faust-to-Python transpilation
step. A comprehensive differentiable programming framework in Faust
would allow DDSP algorithms to be written once, and run anywhere.

Furthemore, The observation has been made that the relative structures
of neural networks and digital filters bear striking similarities
[cite:@kuznetsovDifferentiable2020]. There is a very real sense in
which machine learning problems, and gradient-based optimisation
problems in general, are signal processing problems. The possibility
of turning a signal processing DSL to problems of this sort paves the
way to novel approaches and fresh perspectives on this vibrant field
of research.

Faust is based on a /block-diagram algebra/
[cite:@orlareySyntactical2004]. In this algebra, a given block
diagram, \(D\), is comprised of a primitive block, \(P\), or the
composition of two block diagrams via a composition operator according
to the syntactic rule[fn:1]
#+NAME: eq:syntax
\begin{equation}
D = P \;|\; (D_{1} \diamond D_{2})\;,
\end{equation}
where \(\diamond\) is one of the sequential (:), parallel (,), split (<:),
merge (:>), or recursive (\(\sim\)) composition operators. The creation
of differentiable Faust programs has been shown to be possible by
/undifferentiated/ composition of differentiated primitive blocks
[cite:@rushtonDifferentiable2024], the syntactic rule being
#+NAME: eq:badDiffSyntax
\begin{equation}
\Delta_{N}\llbracket D \rrbracket = \Delta_{N}\llbracket P_{1} \rrbracket \diamond \Delta_{N}\llbracket P_{2} \rrbracket \diamond \dots \diamond \Delta_{N}\llbracket P_{n} \rrbracket\;.
\end{equation}
While effective in facilitating differentiable programming, this
syntax imposed an undesirable verbosity. Our preferred syntactic rule
should be a direct transformation of cref:eq:syntax:
#+NAME: eq:idealDiffSyntax
\begin{equation}
\Delta_{N}\llbracket D \rrbracket
  = \Delta_{N}\llbracket P \rrbracket \;|\; \Delta_{N}\llbracket D_{1} \diamond D_{2} \rrbracket \;.
\end{equation}
The ideal outcome is to permit the programmer to subject an entire
Faust program to automatic differentiation in a single expression,
minimising the amount of re-writing required to ready their code for
transformation. This would render a great proportion of existing Faust
code differentiable, and, in doing so, prepare that wealth of material
for prospective use in optimisation problems, and machine learning
applications. In order to achieve such an outcome, it is necessary to
provide differentiation rules for the five composition operators,
taking into account their arithmetical significance, if any, in a
block diagram to which they contribute.

** Differentiable Block-Diagram Compositions

To define syntactic rules for the derivatives of Faust's composition
operators entails analysing them as if they were themselves primitive
blocks. To do this, we must consider them in tandem with rules for
some of Faust's fundamental primitive operators; these will assist us
in achieving differentiable compositions.

*** Identity Primitive

Faust's identity function, notated \(\_\), describes a connection
between two blocks. Treated as a signal processor, it, adheres to the
following semantic scheme [cite:@orlareySyntactical2004]:
#+NAME: eq:identity
\begin{equation}
  \begin{split}
    \llbracket \_ \rrbracket &\;:\; \mathbb{S} \to \mathbb{S} \\
    \llbracket \_ \rrbracket(s) &= (s)\;.
  \end{split}
\end{equation}
It receives one signal and returns that same signal unchanged,
analogous to multiplying it by a constant signal of value
\(1\). In dual signal arithmetic, this is
#+NAME: eq:dualIdentity
\begin{align}
\langle{}s,\nabla{}s\rangle\langle{}1,\nabla1\rangle
  = \langle{}1s, 1\nabla{}s + s\nabla{}1 \rangle
  = \langle{}s,\nabla{}s\rangle\;,
\end{align}
and the semantic rule for the derivative of the identity function is
thus
#+NAME: eq:diffidentity
\begin{equation}
  \begin{split}
    \Delta_{N}\llbracket \_ \rrbracket &\;:\; \mathbb{S}_{d} \to \mathbb{S}_{d} \\
    \Delta_{N}\llbracket \_ \rrbracket(\langle{}s, \nabla{}s\rangle) &= (\langle{}s, \nabla{}s\rangle)\;.
  \end{split}
\end{equation}
The dual signal \(\langle{}s,\nabla{}s\rangle\) being composed of one
primal signal and \(N\) gradient signals, we can further state that
#+NAME: eq:diffidentity2
\begin{equation}
  \Delta_{N}\llbracket \_ \rrbracket = \big\|_{l=1}^{N+1}\;\_\;.
\end{equation}

*** Parallel Composition

Blocks composed in parallel (see cref:fig:par) do not influence
one-another, arithemtically-speaking. The outputs of the program
\(A,B\), \(y_{1},\dots,y_{o_{A}}\) and \(y_{o_{A}+1},\dots,y_{o_{A}+o_{B}}\), are
essentially the outputs of separate block diagrams; thus
\(\Delta_{N}\llbracket A,B \rrbracket = \Delta_{N}\llbracket A \rrbracket,\Delta_{N}\llbracket
B \rrbracket\), and, more generally
#+NAME: eq:diffpar
\begin{equation}
\Delta_{N}\left\llbracket\big\|_{l=1}^{L}D\right\rrbracket =
  \big\|_{l=1}^{L}\Delta_{N}\llbracket D \rrbracket \;.
\end{equation}

*** Sequential Composition

The application of a function, \(f\), to \(M\) dual signals garners a
definiton of the multivariate chain rule:
#+NAME: eq:dualchain
\begin{align}
\langle{}\mathbf{y},\nabla{}\mathbf{y}\rangle &= f(\langle{}s_{1},\nabla{}s_{1}\rangle,\dots,\langle{}s_{M},\nabla{}s_{M}\rangle) \\
  &= \langle{}f(\mathbf{s}),\nabla(f(\mathbf{s}))\rangle \nonumber \\
  &= \langle{}f(\mathbf{s}),\mathbf{J}_\mathbf{s}\nabla{}f(\mathbf{s})\rangle\;,
\end{align}
where \(\mathbf{J}_{\mathbf{s}}\), the Jacobian matrix of \(\mathbf{s}\),
is an \(M\times{}N\) matrix whose \((k,i)^{\textrm{th}}\) entry is
\(\frac{\partial{}s_{k}}{\partial{}x_{i}}\). In Faust, this is equivalent to differentiating
the sequential composition of \(i_{B} = M\) identity functions with a
block, \(B\), whose inputs are in fact implicit, i.e.:
#+NAME: eq:diffseq1
\begin{equation}
\Delta_{N}\llbracket B \rrbracket = \Delta_{N}\llbracket \big\|_{l=1}^{i_{B}}\,\_ : B \rrbracket\;.
\end{equation}
The differentiable identity function in cref:eq:diffidentity2 gives us
the following equivalence:
#+NAME: eq:diffseq2
\begin{equation}
\big\|_{l=1}^{i_{B}}\big\|_{m=1}^{N+1 }\_ : \Delta_{N}\llbracket B \rrbracket =
  \big\|_{l=1}^{i_{B}}\Delta_{N}\llbracket \_ \rrbracket : \Delta_{N}\llbracket B \rrbracket\;.
\end{equation}
Similarly, the \(o_{A} = M\) outputs of block \(A\) are implicit:
#+NAME: eq:diffseq3
\begin{equation}
\Delta_{N}\llbracket A \rrbracket = \Delta_{N}\llbracket A : \big\|_{l=1}^{o_{A}}\,\_ \rrbracket =
  \Delta_{N}\llbracket A \rrbracket : \big\|_{l=1}^{o_{A}}\Delta_{N}\llbracket\_\rrbracket\;.
\end{equation}
For blocks \(A\) and \(B\) to be composable in sequence (see
cref:fig:seq), \(o_{A}\) must be equal to \(i_{B}\).[fn:2] Since both equal
\(M\), we can conclude that
#+NAME: eq:diffseq4
\begin{align}
\Delta_{N}\llbracket A : B \rrbracket &=
 \Delta_{N}\llbracket A \rrbracket : \big\|_{l=1}^{M}\Delta_{N}\llbracket\_\rrbracket : \Delta_{N}\llbracket B \rrbracket \nonumber \\
 &= \Delta_{N}\llbracket A \rrbracket : \Delta_{N}\llbracket B \rrbracket,\quad \forall A,B,\; o_{A} = i_{B}\;.
\end{align}

*** Split Composition

Composing \(A\) and \(B\), \(i_{B} = ko_{A}\), with the split
composition operator (see cref:fig:split) duplicates the output
signals of \(A\), \(k\) times, passing each group of duplicates to
\(B\), in order:
#+NAME: eq:split
\begin{equation}
A <: B = A : \big\|_{l=1}^{o_{A}}\,\__{\,l} <: \big\|_{m=1}^{k}\big\|_{l=1}^{o_{A}}\,\__{\,l} : B\;.
\end{equation}
Cref:eq:split indicates that
#+NAME: eq:
\begin{gather}
D : \big\|_{l=1}^{o_{D}}\,\_ = \big\|_{l=1}^{i_{D}}\,\_ : D = D\;,
\end{gather}
and, differentiating
#+NAME: eq:
\begin{gather}
\Delta_{N}\llbracket D \rrbracket : \big\|_{l=1}^{o_{D}}\Delta_{N}\llbracket \_ \rrbracket = \Delta_{N}\llbracket D \rrbracket \\
\big\|_{l=1}^{i_{D}}\Delta_{N}\llbracket \_ \rrbracket : \Delta_{N}\llbracket D \rrbracket = \Delta_{N}\llbracket D \rrbracket\;.
\end{gather}
Since split composition maintains signal order within each group,
applying derivative transformation to both sides of cref:eq:split,
gives
#+NAME: eq:diffsplit
\begin{align}
&\quad\;\Delta_{N}\llbracket A <: B \rrbracket \nonumber\\
  &= \Delta_{N}\llbracket A \rrbracket :
    \big\|_{l=1}^{o_{A}}\Delta_{N}\llbracket \__{\,l} \rrbracket <:
    \big\|_{m=1}^{k}\big\|_{l=1}^{o_{A}}\Delta_{N}\llbracket \__{\,l} \rrbracket :
    \Delta_{N}\llbracket B \rrbracket \nonumber \\
  &= \Delta_{N}\llbracket A \rrbracket <: \Delta_{N}\llbracket B \rrbracket \;.
\end{align}

*** Cut Primitive
:PROPERTIES:
:CUSTOM_ID: sec:cut
:END:

The cut function (!) terminates a signal, and combinations of the cut
and identity functions can be used to create complex signal routings.
For instance, to interleave two groups of two signals, such that
\(s_{1,1},s_{1,2},s_{2,1},s_{2,2}\) becomes \(s_{1,1},s_{2,1},s_{1,2},s_{2,2}\), one may write
#+NAME: eq:interleave22
\begin{equation}
\_,\_,\_,\_ \;<:\; \_,!,\_,!,!,\_,!,\_\;.
\end{equation}
To differentiate an expression such as this, the appropriate
accompaniment to cref:eq:diffidentity2,eq:diffsplit for the derivative
of the cut function is
#+NAME: eq:cut
\begin{equation}
  \begin{split}
    \Delta_{N}\llbracket ! \rrbracket &= \big\|_{l=1}^{N+1}!\;.
  \end{split}
\end{equation}
A general interleaving function, as used in cref:lst:diffAdd, can be
written using the cut and identity primitives, plus parallel and split
composition (see cref:lst:interleave). Comprising only differentiable
elements, it too is differentiable.
#+name: lst:interleave
#+caption: A Faust function that interleaves \(c\) groups of \(r\)
#+caption: signals into \(r\) groups of \(c\) signals. See cref:eq:interleave.
#+attr_latex: :options float=ht
#+begin_src faust :noeval
interleave(r, c) = par(l, r * c, _)
  <: par(m, r, par(n, c,
       cutpq(m, r - m - 1)))
with {
  cutpq(0, q) = _,par(l, q, !);
  cutpq(p, 0) = par(l, p, !),_;
  cutpq(p, q) =
    par(l, p, !),cutpq(0, q);
};
#+end_src

*** Delay Primitive
:PROPERTIES:
:CUSTOM_ID: sec:delay
:END:

Faust provides two primitives for delaying signals; here we will focus
on the ~mem~ primitive, which, when composed in sequence with a
signal, applies a one-sample delay.  In general, a fixed delay can be
represented as the convolution of a signal, \(s(x,t)\), with a
time-shifted impulse,
#+NAME: eq:delayAsConv
\begin{equation}
s(x, t-\tau) = s(x, t) \ast h(t)\;,\quad h(t) = \delta(t - \tau)\;.
\end{equation}
Taking the temporal-parametric Fourier transform of the derivative of
the convolution with respect to \(x\), then using the convolution and
differentiation properties of the transform
[cite:@mcclellanSignal2003], yields
#+NAME: eq:timeinvariantdiff
\begin{align}
&\quad\;\mathcal{F}\left[\frac{\partial{}}{\partial{}x}\left(s(x,t) \ast h(t)\right)\right] \nonumber\\
  &= j\omega_{x}S(j\omega_{x},j\omega_{t})H(j\omega_{t}) \\
  &= \mathcal{F}\left[\frac{\partial{}}{\partial{}x}s(x,t)\right]\cdot\mathcal{F}\left[h(t)\right]\;,
\end{align}
which demontsrates that differentiation is time-invariant, and allows
us to deduce the convolution rule of differentiation --- to
differentiate a convolution we need only differentiate one of the
functions being convolved. Applied to dual numbers,
#+NAME: eq:
\begin{equation}
\langle{}s,\nabla{}s\rangle\ast\langle{}h,\nabla{}h\rangle = \langle{}s\ast{}h, \nabla{}s\ast{}h\rangle\;,
\end{equation}
giving us the following semantic scheme:
#+NAME: eq:diffmem
\begin{equation}
  \begin{split}
    \Delta_{N}\llbracket\texttt{mem}\rrbracket &\;:\; \mathbb{S}_{d} \to \mathbb{S}_{d} \\
    \Delta_{N}\llbracket\texttt{mem}\rrbracket(\langle{}s, \nabla{}s\rangle) &= (\langle{}y, \nabla{}y\rangle) \\
    \langle{}y(t),\nabla{}y(t)\rangle &= \langle{}s(t-1),\nabla{}s(t-1)\rangle\;.
  \end{split}
\end{equation}

*** Recursive Composition

The differentiability of recursive filters is well-established
[cite:@shynkAdaptive1989] --- the derivative of a recursion is itself
recursive. To establish a differentiation rule for Faust's recursive
composition operator (see cref:fig:rec), we can expand upon
cref:eq:dualchain, introducing an inner function, \(g\), that acts
upon a vector of past output signals:
#+NAME: eq:dualrec
\begin{equation}
\begin{split}
\langle{}\mathbf{y}(t),\,&\mathbf{J}_{\mathbf{y}}(t)\rangle = \bigg\langle{}f(\mathbf{s}(t),g(\mathbf{y}(t-1))), \\
  &\mathbf{J}_{\mathbf{s}}(t)\frac{\partial{}f}{\partial{}\mathbf{s}(t)} +
  \mathbf{J}_{\mathbf{y}}(t-1)\nabla{}g(\mathbf{y}(t-1))\frac{\partial{}f}{\partial{}g}\bigg\rangle\;.
\end{split}
\end{equation}
For the case of a one-pole filter, this reduces to
#+NAME: eq:one-pole
\begin{equation}
\langle{}y(t),\nabla{}y(t)\rangle = \langle{}s(t) + y(t-1),\nabla{}s(t) + \nabla{}y(t-1)}\rangle\;,
\end{equation}
which, comparing with cref:eq:diffmem, could just as readily be the
end of the semantic scheme for the derivative of an impossible
/unrolled/ recursion:
#+NAME: eq:recSemantic
\begin{equation}
\Delta_{N}\llbracket \_,\texttt{mem} : + \rrbracket(\langle{}s,\nabla{}s\rangle,\langle{}y,\nabla{}y\rangle) = (\langle{}y,\nabla{}y\rangle)\;.
\end{equation}
This points to the fact that Faust's recursive composition operator
introduces an implicit one sample delay, and acts like /sequential
composition through time/, thus:
#+NAME: eq:diffrec
\begin{equation}
\Delta_{N}\llbracket A \sim B \rrbracket = \Delta_{N}\llbracket A \rrbracket \sim \Delta_{N}\llbracket B \rrbracket
\end{equation}

*** Merge Composition

Merge composition (cref:fig:merge) entails summation
[cite:@orlareySyntactical2004], thus the addition rule of
differentiation must be obeyed when merging dual signals. A merge \(A
:> B\), can be viewed as an interleaving of the outputs of \(A\) into
\(i_{B}\) groups of \(k=o_{A}/i_{B}\) signals, followed by a merging of each
group into one signal path:
#+NAME: eq:expandedMerge
\begin{equation}
A :> B = A : \chi(i_{B},k) : \big\|_{l=1}^{i_{B}}\left(
  \big\|_{m=1}^{k}\,\_ :> \_
\right) : B\;.
\end{equation}
Clearly \(\Delta_{N}\llbracket A:>B \rrbracket\) cannot be defined
recursively, however. Note that for \(i_{B}=1\) the right hand side of
cref:eq:expandedMerge reduces to \(A:>\_\), essentially the term in
parenthesis, which can be handled thanks to the definitions in
cref:eq:diffidentity2,eq:diffpar,
#+NAME: eq:mergeOntoOne
\begin{equation}
\Delta_{N}\llbracket \big\|_{m=1}^{k}\,\_ :> \_ \rrbracket
  = \big\|_{m=1}^{k}\Delta_{N}\llbracket \_ \rrbracket :> \Delta_{N}\llbracket \_ \rrbracket\;.
\end{equation}
The order of signals is maintained, so the resulting dual signal is
guaranteed to be of the form
\(\left\langle{}\sum_{m=1}^{k}s_{m},\sum_{l=m}^{k}\nabla{}s_{m}\right\rangle\).
For the full expression for differentiable merge composition, see
cref:eq:diffmerge.

** Summary

The rules for differentiating Faust's composition operators are collected
in cref:fig:diffRules. With the implementation of these rules via
pattern matching, Faust programs can be recursively decomposed
according to their composition operators, and re-composed in
differentiable fashion. For example:
#+NAME: eq:diffComposition
\begin{equation}
\begin{split}
\Delta_{N}\llbracket (A,C):B \rrbracket
  &= \Delta_{N}\llbracket A,C \rrbracket : \Delta_{N}\llbracket B \rrbracket \\
  &= \Delta_{N}\llbracket A \rrbracket , \Delta_{N}\llbracket C \rrbracket : \Delta_{N}\llbracket B \rrbracket\;.
\end{split}
\end{equation}
Moreover, programs consisting of differentiable primitives, such as
~mem~ (cref:sec:delay) or the addition operator (cref:lst:diffAdd),
can be automatically differentiated to arbitrary order,
\(\Delta_{N}\llbracket{\scriptstyle\Delta_{N}\llbracket
\cdots}{\scriptscriptstyle\Delta_{N}\llbracket \cdot \rrbracket}{\scriptstyle \cdots
\rrbracket}\rrbracket\).

*** Limitations

Although the described scheme renders a great deal of existing Faust
code differentiable, certain primitives and language constructs
cannot, at present, be subjected to AD.

**** Route primitive
:PROPERTIES:
:UNNUMBERED:
:END:
Faust's ~route~ primitive cannot be pattern-matched, which precludes
the derivative transformation of arbitrary signal routing expressions
based on that primitive, such as those in the =routes.lib=
library.[fn:4] As described in cref:sec:cut, however, such expressions
can be recreated via judicious use of the /identity/ and /cut/
primitives, and common routing tasks such as interleaving can be
achieved differentiably with code such as in cref:lst:interleave.

**** User interface primitives
:PROPERTIES:
:UNNUMBERED:
:END:
The obvious derivative transformation of a user interface element is
to an input variable, i.e. a parameter for optimisation such as a
weight in a neural network. Input variables must be identifiable so as
create the appropriate corresponding dual signal
\(\langle{}x_{i},\begin{bmatrix}0 & \cdots & \frac{\partial{}x_{i}}{\partial{}x_{i}} & \cdots & 0\end{bmatrix}^{T}\rangle\).

Though matchable in principle, to perform a derivative transformation
on a user interface element it is necessary to pattern-match on all of
its parameters. For instance, to transform the slider ~hslider("x1", 0,
-1, 1, .01)~, whose parameters are a string identifier, inital value,
minimum, maximum and step, it is necessary to write ~diff(hslider("x1",
0, -1, 1, .01))~. Clearly, in the general case, this is not practical.

The approach taken at the time of writing has been to move input
variables outside of the general AD transformation, with a wrapper
function used to indicate the number of input signals (whose gradients
are zero vectors) and input variables received by the main body of the
differentiated block diagram. Though imposing a certiain coding
convention, we consider this comparable with the particular
requirements imposed by PyTorch, TensorFlow, and JAX, with their
respective dedicated datatypes for learnable parameters.
