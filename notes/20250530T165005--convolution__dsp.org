I want to run real-time impulse response convolutions on an embedded
platform that's programmed in C/C++. Let's explore how I should go
about implementing that.

* Notation

Discrete time-domain convolution of a signal \(s[n]\) with an impulse response
\(h[n]\) is notated [cite:@mcclellanSignal2003 ch. 5]

\begin{equation}
y[n] = s[n] \ast h[n] = \sum^{M}_{k=0} h[k]s[n-k]
\end{equation}

* Example

Let's consider an example where \(s[n]\) is a rectangular pulse,
\(s[n] = \{1,1,1,1\}\) and \(h[n]\), with length \(M = 3\), is an
inverted ramp, \(h[n] = \{1, 0.5, 0.25\}\). The convolution can be
represented in tabular fashion, demonstrating /synthetic polynomial
multiplication/.

#+attr_latex: :scale 2
\begin{equation}
\begin{array}{r||c|cccccc|c}
n          & n<0 & 0 & 1   & 2    & 3    & 4    & 5    & n>5 \\
\hline \hline
s[n]       & 0   & 1 & 1   & 1    & 1    & 0    & 0    & 0 \\
h[n]       & 0   & 1 & 0.5 & 0.25 &      &      &      &   \\
\hline
h[0]s[n]   & 0   & 1 & 1   & 1    & 1    & 0    & 0    & 0 \\
h[1]s[n-1] & 0   & 0 & 0.5 & 0.5  & 0.5  & 0.5  & 0    & 0 \\
h[2]s[n-2] & 0   & 0 & 0   & 0.25 & 0.25 & 0.25 & 0.25 & 0 \\
\hline
y[n]       & 0   & 1 & 1.5 & 1.75 & 1.75 & 0.75 & 0.25 & 0
\end{array}
\end{equation}

* Code

The above is fine, but in practice it's not going to be possible to
lean on a library implementation. Let's perform the calculation by
hand.

#+begin_src julia :session
function convolve(s, h)
    M = length(h) - 1
    len = length(s) + M
    y = OffsetArray(zeros(len), -1)
    for n = 0:len
        for k = 0:M
            if n - k >= 0 && n - k < length(s)
                y[n] += h[k] * s[n - k]
            end
        end
    end
    return y
end;

y2 = convolve(s, h)
#+end_src

* Computational Requirements

Clearly then, the longer the impulse response, the heavier the
calculation --- which is completely logical, since we're essentially
applying an FIR filter (with three taps in this case). From /Spectral
Audio Signal Processing/ [cite:@smithSpectral2011, ch. 8.1.1]:

#+begin_quote
...the computational complexity of direct form cyclic convolution of
two N-point signals is \(\mathcal{O}(N^{2})\) ... More precisely, [it]
requires \(N^{2}\) multiplies and \(N(N-1)\) additions...
#+end_quote

* FFT-based Convolution

Time-domain convolution's frequency-domain counterpart is
multiplication

\begin{equation}
\mathrm{DFT}_{k}\{h \ast x\} = H(\omega_{k})X(\omega_{k})\;,
\end{equation}

and it is of complexity \(\mathcal{O}(N\log_{2}N)\), performing optimally
when the length of the FFT is a power of two.

#+begin_src julia :session
Pkg.add("FFTW")
using FFTW

nfft = nextpow(2, max(length(s), length(h)))
S = fft([collect(s); zeros(nfft - length(s))])
H = fft([collect(h); zeros(nfft - length(h))])
Y = S.*H
#+end_src

** Convolving Long Signals

[[https://dvcs.w3.org/hg/audio/raw-file/tip/webaudio/convolution.html][However, for long IRs...]]

#+begin_quote
...naively doing a standard overlap-add FFT convolution using an FFT
of size N with L=N/2, where N is chosen to be at least twice the
length of the convolution kernel (zero-padding the kernel) to perform
each convolution operation...  would incur a substantial input to
output pipeline latency on the order of L samples. Because of the
enormous audible delay, this simple method cannot be used. Aside from
the enormous delay, the size N of the FFT could be extremely
large. For example, with an impulse response of 10 seconds at 44.1Khz,
N would equal 1048576 (2^20). This would take a very long time to
evaluate.
#+end_quote

Essentially, for block-based processing, it's going to be necessary to
break the FFT of the impulse response into chunks too. Enter
overlap-add (and/or overlap-save).

For each input block, convolved with the IR, the
corresponding output block will be of length ~nFFT~. Having computed an
output block, it should be overlap-added to the existing output
signal; the second output block should combined with the tail of the
first output block, i.e. the value of the output sample \(y(M+m)\)
will be the sum of the first sample of the second output block and the
\((M+m)^{\mathrm{th}}\) sample of the first output block.

    Sm = fft(szp)
    Y7 = Sm .* H3_1
    y7[m:m+nFFT-1] += real(ifft(Y7))
end
#+end_src

*** Chopping up the IR too

So, now there case where \(h\) is very long. According to Claude🤖,
the next step is /partitioned convolution/ (not covered by Smith
directly, but sort of included as an exercise to the reader). The
process:

- Split \(h\) into \(K\) blocks of length \(M\): \(h_{0}, h_{1}, \dots, h_{K-1}\).
- For each input block \(x_{n}\), convolve with each partition of \(h\).
- The output block \(y_{n}\) is the sum of each of the convolutions of
  the \((n-k)^{\mathrm{th}}\) input blocks with the \(k^{\mathrm{th}}\)
  partition of \(h\).

(I guess overlap-add will be necessary too. In any case)

\begin{equation}
y_{n} = x_{n} \ast h = \sum_{k=0}^{K-1} x_{n-k} \ast h_{k}
\end{equation}

E.g., for \(K = 3\)

\begin{align}
y_{0} &= x_{0} \ast h_{0} \\
y_{1} &= x_{1} \ast h_{0} + x_{0} \ast h_{1} \\
y_{2} &= x_{2} \ast h_{0} + x_{1} \ast h_{1} + x_{0} \ast h_{2} \\
y_{3} &= x_{3} \ast h_{0} + x_{2} \ast h_{1} + x_{1} \ast h_{2} \\
&\dots
\end{align}

So, one FFT, \(X_{n}\), for each input block, multiplied by each of
\(H_{k}\)...

\begin{equation}
y_{n} = \sum_{k=0}^{K-1} \Re\left(\mathcal{F}^{-1}\left\{X_{n-k} \cdot H_{k}\right\}\right)
\end{equation}

So (for uniform partitioning) it's probably going to be necessary to keep a
buffer \(\mathbf{X}\), a matrix composed of \(K\) FFT vectors, \(X_{n}\).

**** Non-uniform partitioning

Interesting is the idea of /non-uniform partitioning/. Later partitions
can be progressively larger powers of two, exploiting the fact that
longer FFTs, while introducing more latency, are more computationally
efficient than shorter ones (which may in fact be /less/ efficient than
using direct form time-domain) plus the idea that the latter part of
the IR is typically of less significance to the output (though what
the nature of that (in)significance is I can't say without digging
into the literature myself). The aim (echoed [[https://dvcs.w3.org/hg/audio/raw-file/tip/webaudio/convolution.html][here]], so to speak) is to
minimise latency by having output samples ready to go as close to
immediately as possible, perhaps even using direct form for the first
couple of blocks.

Now perform the partitioned convolution. 

#+begin_src julia :session fupola :results file graphics :file (format "%s/fupola.gif" (org-babel-temp-directory)) :var dir=(org-babel-temp-directory)
# An output vector long enough to store all the samples that will be
# produced.
y = zeros(length(szp) + nB)

anim = @animate for m=1:numBlocks
    # Zero-pad the current input block.
    local szp = [sBlocks[:,m]; zeros(nFFT - nB)]
    S = fft(szp)

    # Get the convBuf block index
    block = mod1(m, numBlocks)#blocksPerIR)
    
    # Store the convolution of the input block with each IR partition.
    for l=1:size(H, 1)
        convBuf[block, l, :] = real(ifft(S .* H[l]))
    end

    # Get the starting sample index at which to write to the output
    # vector.
    yStart = 1 + (m - 1) * size(sBlocks, 1)

    # OK, now we need to iterate as far back as to cover all the
    # partitions with a commensurate number of blocks...
    # For a start we need samples [1:nBlock] from the convolution of
    # the current block with the first partition.
    # Then we need samples [nBlock+1:2*nBlock] from the convolution of
    # the previous block with... well, probably the first partition
    # again...
    for l=1:blocksPerIR
        block = mod1(m - (l - 1), numBlocks)#blocksPerIR)
        partition = Integer(floor((l - 1) / blocksPerPartition)) + 1
        offset = 1 + (mod1(l, blocksPerPartition) - 1) * nB

        # Overlap-add.
        y[yStart:yStart + nB - 1] += convBuf[
            block,
            partition,
            offset:offset + nB - 1
        ]

        # Ideally this shouldn't be necessary, but frankly this isn't
        # my finest work.
        if mod(l, blocksPerPartition) == 0
            y[yStart + nB:yStart + (2*nB) - 1] += convBuf[
                block,
                partition,
                offset + nB:offset + (2*nB)-1
            ]
        end
    end
    
    plot(y, title="FUPOLA", label=@sprintf("y[n], block %d", m))
end

plot(y, title="FUPOLA", label="y[n]")
savefig("$dir/fupola")

*** Overlap-save

Since overlap-add didn't turn out very nicely, and since overlap-save
is apparently the more common approach\nbsp[cite:@wefersPartitioned2015],
let's give the latter a try. First of all, (frequency-domain)
uniformly partitioned (FUPOLS).

Setup is as it was for FUPOLA.

#+begin_src julia :session fupols :results output
using Plots, Printf, FFTW, DSP

# Input signal.
Ns = 9600
nTaps = 7
s = zeros(Ns)
for n=0:nTaps-1
    s[1 + Integer(n * floor(Ns / nTaps))] = .5 * (-1)^n
end

# Impulse response.
Nh = Integer(Ns/4)
nTaps = 17
h = zeros(Nh)
for n=0:nTaps-1
    h[1 + Integer(n * floor(Nh/nTaps))] = .5 * exp(-.25 * n) * (-1)^(n+1)
end

# Block length
B = 128
# Partition length
L = B
# Transform size
K = 2 * B
# Partition count
P = Integer(ceil(Nh/L))

@printf "Block length %d | Partition length %d | Transform size %d | Partition count %d" B L K P
#+end_src

#+RESULTS:
: Block length 128 | Partition length 128 | Transform size 256 | Partition count 19

Zero-pad, as before.

#+begin_src julia :session fupols :results output
hzp = [h; zeros(P * L - length(h))]
hp = reshape(hzp, L, P)
szp = [s; zeros(length(hzp) - B)]
# Number of blocks in the input signal.
M = length(szp) / B
if isinteger(M)
    M = Integer(M)
    sb = reshape(szp, B, M)
    @printf "Input is of shape %s; IR is of shape %s." size(sb) size(hp)
else
    @printf "Somehow, the number of blocks (M) is not an integer; are you sure about that?"
end
#+end_src

#+RESULTS:
: Input is of shape (128, 93); IR is of shape (128, 19).

Split the length \(N_{h}\) IR into \(P = \lceil N/L \rceil\) length-\(L\)
sub-filters. Zero-pad each sub-filter to length \(2L\) and perform a
\(2L\)-point FFT on each.

#+begin_src julia :session fupols :results output
H = [fft([hp[:,i]; zeros(L)]) for i in 1:P]
@printf("H contains a %d-point FFT of each of the %d IR partitions.",
        size(H[1], 1), size(H, 1))
#+end_src

#+RESULTS:
: H contains a 256-point FFT of each of the 19 IR partitions.

Now for the FUPOLS algorithm.

#+begin_src julia :session fupols :results file graphics :file (format "%s/fupols.gif" (org-babel-temp-directory)) :var dir=(org-babel-temp-directory)
y = zeros(length(szp) + B)

# The input buffer acts as a 2B-point sliding window of the input
# signal.
inputBuffer = zeros(2 * B)

# Frequency-domain delay line.
FDL = [zeros(ComplexF64, K) for i in 1:P]

anim = @animate for m=1:M
    # With each new input block, the right half of the input buffer is
    # shifted to the left and the new block is stored in the right
    # half.
    inputBuffer[1:B] = inputBuffer[B + 1:2 * B]
    inputBuffer[B + 1:2 * B] = sb[:,m]

    # All contents in the FDL are shifted up by one slot.
    FDL[2:P] = FDL[1:P - 1]
    # A 2B-point FFT is computed from the input buffer, resulting in
    # B+1 complex-conjugate symmetric DFT coefficients. The result is
    # stored in the first slot of the FDL.
    FDL[1] = fft(inputBuffer)

    X = zeros(ComplexF64, K)
    
    # The P sub-filter spectra are pairwise multipiled with the input
    # spectra in the FDL. The results are accumulated in the frequency
    # domain.
    for i=1:P
        X += FDL[i] .* H[i]
    end

    # Of the accumulated spectral convolutions, a 2B-point IFFT is
    # computed.
    x = real(ifft(X))
    # From the resulting 2B samples, the left half is discarded and the
    # right half is returned as the next output block.
    y[1 + (m - 1) * B:(m - 1) * B + B] = x[B + 1:2 * B]

    plot(y, title="FUPOLS", label=@sprintf("y[n], block %d", m))
end

gif(anim, "$dir/fupols.gif", fps=10)
#+end_src
